# 并行 K-means 算法设计小白汇报版

## 1. 我们这个项目在做什么

这个项目实现的是 K-means 聚类算法。简单来说，K-means 的任务是:

```text
把很多数据点自动分成 K 类，让同一类里的点尽量相似，不同类之间尽量分开。
```

例如有很多二维点、多维样本或用户特征，K-means 会尝试把它们分成若干组。每一组都有一个“中心点”，样本会被分配给离自己最近的中心。

本项目一共实现了三个版本:

| 版本 | 含义 |
| --- | --- |
| `serial` | 串行版本，只用一个线程，一个一个样本处理 |
| `parallel` | 朴素 OpenMP 并行版本，把样本分给多个线程同时处理 |
| `optimized` | 优化并行版本，在并行基础上加入 Hamerly 剪枝、SoA 数据布局和 padding |

这三个版本的目标是一样的，都是做 K-means 聚类；区别在于计算速度和优化方法不同。

## 2. K-means 的基本过程

K-means 可以理解为重复做两件事:

1. 分配样本: 每个样本找到离自己最近的聚类中心。
2. 更新中心: 每一类重新计算平均值，得到新的聚类中心。

这两步会反复进行，直到聚类中心基本不动，或者达到最大迭代次数。

伪代码如下:

```text
初始化 K 个聚类中心

重复执行:
    对每个样本:
        计算它到所有中心的距离
        分配给最近的中心

    对每个聚类:
        根据该类所有样本重新计算中心

    如果中心移动很小，则停止
```

其中最耗时的一步是:

```text
计算每个样本到所有聚类中心的距离
```

因为如果样本很多、维度很高、聚类数量也多，那么距离计算次数会非常大。

## 3. 串行版本 serial 是怎么做的

串行版本最直观:

```text
一个线程从第 1 个样本算到第 N 个样本。
```

对于每个样本，它会计算这个样本到所有 `K` 个中心的距离，然后找到最近中心。

优点:

1. 逻辑简单。
2. 结果稳定。
3. 适合作为实验基准。

缺点:

1. 只使用一个 CPU 核心。
2. 样本数量大时速度慢。

所以串行版本主要用来作为对照组，也就是计算加速比时的基准。

## 4. 为什么 K-means 适合并行

在“样本分配”阶段，每个样本找最近中心时，样本之间互不影响。

例如:

```text
样本 1 找最近中心，不需要知道样本 2 的计算结果。
样本 2 找最近中心，也不需要等待样本 1。
```

这说明样本分配阶段天然适合并行。

我们可以把样本分成几份:

```text
线程 1 处理一部分样本
线程 2 处理一部分样本
线程 3 处理一部分样本
...
```

这样多个线程同时计算，整体时间就会下降。

## 5. 朴素并行版本 parallel 的设计

朴素并行版本使用 OpenMP 实现。OpenMP 可以让 C++ 程序比较方便地使用多线程。

核心思想:

```text
把样本循环并行化。
```

也就是原来串行版本是:

```cpp
for 每个样本:
    找最近中心
```

并行版本变成:

```cpp
#pragma omp parallel
多个线程同时处理不同样本
```

### 5.1 为什么不能所有线程直接写全局 sum

更新聚类中心时，需要统计每一类的样本和:

```text
sum[c]    第 c 类所有样本的坐标和
count[c]  第 c 类样本数量
```

如果多个线程同时写同一个 `sum[c]` 或 `count[c]`，就会产生数据竞争。

举个例子:

```text
线程 1 想把 count[0] 加 1
线程 2 也想把 count[0] 加 1
如果它们同时写，结果可能只加了一次
```

这就是并行程序里常见的数据竞争问题。

### 5.2 我们怎么避免数据竞争

我们的做法是:

```text
每个线程使用自己的 local_sum 和 local_count。
```

也就是说:

```text
线程 1 写 local_sum[1]
线程 2 写 local_sum[2]
线程 3 写 local_sum[3]
```

每个线程只写自己的数组，不抢同一个变量。

等所有线程算完后，再统一合并:

```text
global_sum = local_sum[1] + local_sum[2] + ...
global_count = local_count[1] + local_count[2] + ...
```

这样就避免了在主样本循环中使用 `atomic` 或 `critical`，减少同步开销。

## 6. optimized 版本的总体思路

朴素并行版本只是“让多个线程一起算”，但每个样本仍然要计算到所有中心的距离。

optimized 版本继续优化，核心目标是:

```text
能不算的距离就不算，必须算的距离算得更快，多线程写数据时尽量不互相干扰。
```

它包含三个创新点:

1. Hamerly 三角不等式剪枝。
2. SoA 数据布局优化。
3. padding 消除 false sharing。

下面逐个解释。

## 7. 创新点一: Hamerly 三角不等式剪枝

### 7.1 它解决什么问题

普通 K-means 每一轮都会让每个样本重新计算到所有中心的距离。

但是实际迭代到后面时，很多样本所属类别已经很稳定了。也就是说:

```text
这个样本上一轮属于第 3 类，这一轮大概率还是第 3 类。
```

如果我们能提前判断它不会换类别，就可以跳过完整距离计算。

### 7.2 Hamerly 剪枝维护什么信息

对每个样本，我们维护三个信息:

```text
label[i]  样本 i 当前属于哪个中心
upper[i]  样本 i 到当前所属中心距离的上界
lower[i]  样本 i 到其它中心距离的下界
```

对每个中心，我们还计算:

```text
s[c] = 0.5 * 最近其它中心到中心 c 的距离
```

可以理解为:

```text
s[c] 表示中心 c 周围的一个安全范围。
```

### 7.3 怎么判断可以跳过

判断条件是:

```text
upper[i] <= max(lower[i], s[label[i]])
```

如果这个条件成立，说明样本 i 当前所属中心已经足够近，其它中心不太可能比它更近。

于是:

```text
直接跳过这个样本到所有中心的完整距离计算。
```

如果条件不成立，再老老实实重新计算它到所有中心的距离。

### 7.4 中心移动后怎么办

每轮中心都会移动，所以之前的上下界需要更新。

设:

```text
move[c]   第 c 个中心移动距离
max_move  所有中心中最大的移动距离
```

更新方式:

```text
upper[i] += move[label[i]]
lower[i] = max(0, lower[i] - max_move)
```

这一步保证下一轮剪枝判断仍然是安全的。

### 7.5 剪枝效果怎么看

optimized 版本会输出:

```text
distance_calculations  实际做了多少次距离计算
skipped_points         跳过了多少次样本完整距离计算
pruning_rate           剪枝率
```

剪枝率越高，说明越多样本没有重新计算所有中心距离，优化效果越明显。

## 8. 创新点二: SoA 数据布局优化

### 8.1 AoS 是什么

原始数据布局是 AoS，可以理解为:

```text
一个样本的所有维度放在一起。
```

例如:

```text
样本 1: x1, y1, z1
样本 2: x2, y2, z2
样本 3: x3, y3, z3
```

在代码里是:

```text
data_aos[i * dim + d]
```

这种方式直观，适合按样本读取。

### 8.2 SoA 是什么

SoA 是把同一维度的数据放在一起:

```text
第 1 维: x1, x2, x3, ...
第 2 维: y1, y2, y3, ...
第 3 维: z1, z2, z3, ...
```

在代码里是:

```text
data_soa[d * n + i]
```

### 8.3 为什么 SoA 可能更快

CPU 读取内存时喜欢连续访问。如果数据连续，缓存命中率更高，也更容易被编译器自动向量化。

optimized 版本使用 SoA 布局做距离计算，同时把中心也按维度优先存储。这样在距离计算中，内层循环访问的中心数据更加连续。

简单理解:

```text
SoA 让 CPU 更容易一口气读一排连续数据。
```

这有助于缓存访问和 SIMD 自动向量化。

## 9. 创新点三: padding 消除 false sharing

### 9.1 false sharing 是什么

CPU 缓存不是一个变量一个变量管理的，而是一整块一整块管理的，这一块通常叫 cache line。

如果两个线程写的是不同变量，但这两个变量刚好在同一个 cache line 里，就会出现 false sharing。

现象是:

```text
线程 1 写自己的变量
线程 2 写自己的变量
看起来互不冲突
但 CPU 缓存层面它们在抢同一个 cache line
```

这会导致缓存行反复失效，降低并行性能。

### 9.2 我们怎么处理

optimized 版本给每个线程的局部统计结构做 64 字节对齐:

```cpp
struct alignas(64) ThreadLocalStats {
    ...
};
```

同时局部数组也按 cache line 做 padding。

这样可以尽量保证:

```text
不同线程写的数据不要挤在同一个 cache line 里。
```

主样本循环中，每个线程只写自己的 `local_sum` 和 `local_count`，最后统一合并，所以没有在主循环中使用 `atomic` 或 `critical`。

## 10. 三个版本的对比

| 版本 | 主要特点 | 优点 | 缺点 |
| --- | --- | --- | --- |
| serial | 单线程逐个样本计算 | 简单稳定，适合作基准 | 速度慢 |
| parallel | 多线程处理样本 | 能利用多核 CPU | 仍然计算所有距离 |
| optimized | 剪枝 + SoA + padding | 距离计算更少，内存访问更友好，并行冲突更少 | 实现更复杂，额外维护 upper/lower |

## 11. 实验结果怎么解释

实验中可以重点看这几个指标:

| 指标 | 含义 |
| --- | --- |
| time | 算法运行时间 |
| iterations | 实际迭代次数 |
| inertia | 最终 SSE，越小说明聚类目标函数越小 |
| distance_calculations | 实际距离计算次数 |
| skipped_points | Hamerly 剪枝跳过次数 |
| pruning_rate | 剪枝率 |
| speedup | 加速比 |
| efficiency | 并行效率 |

如果 `serial`、`parallel`、`optimized` 的 inertia 基本一致，说明它们得到的聚类结果是可比的。

如果 optimized 的 `pruning_rate` 很高，说明 Hamerly 剪枝确实跳过了大量计算。

如果 parallel 和 optimized 的运行时间明显低于 serial，说明并行化和优化是有效的。

## 12. 汇报时可以这样总结

可以用下面这段作为汇报总结:

```text
本项目首先实现了标准串行 K-means，并以此作为性能基准。
然后使用 OpenMP 对样本分配阶段进行并行化，每个线程使用局部 sum 和 count，避免数据竞争。
在朴素并行基础上，进一步实现 optimized 版本:
第一，利用 Hamerly 三角不等式剪枝减少不必要的距离计算；

第三，通过 cache line padding 减少 false sharing。
实验结果表明，optimized 版本在保持聚类结果一致的前提下，显著减少距离计算次数并降低运行时间。
```
